Panorama Protocoles · 1/2

REST / HTTP — Le standard universel

Modèle de communication
  • Requête/Réponse synchrone — client initie, serveur répond
  • Format JSON — texte UTF-8, lisible humain, verbeux
  • HTTP/1.1 — une connexion par requête ou keep-alive
  • Stateless — chaque requête porte ses propres informations
  • Verbes HTTP — GET, POST, PUT, DELETE, PATCH
Forces
  • Universellement supporté (navigateurs, mobiles, IoT)
  • Outillage mature : Postman, curl, Swagger/OpenAPI
  • Debugging facile — texte lisible dans Wireshark
  • Onboarding rapide — courbe d'apprentissage faible
  • Firewall & proxy friendy sans configuration
Faiblesses
  • JSON verbeux — headers répétés à chaque requête
  • Pas de contrat fort (OpenAPI est optionnel)
  • Streaming difficile (SSE ou WebSocket requis)
  • Head-of-line blocking (HTTP/1.1)
  • Pas de génération de code client natif
Format d'un payload REST
POST /sensors/readings HTTP/1.1
Content-Type: application/json
Authorization: Bearer ...

{
  "sensor_id": "T-001",
  "temperature": 72.4,
  "timestamp": 1716000000
}
// ~260B body + ~150B headers = ~410B/req
Panorama Protocoles · 2/2

gRPC & Alternatives

gRPC — Protocol Buffers + HTTP/2
Forces
  • Payload compact (binaire Protobuf)
  • HTTP/2 : multiplexage, HPACK
  • Streaming natif (4 modes)
  • Contrat fort (.proto)
  • Code client généré (protoc)
Faiblesses
  • protoc requis pour builds
  • Pas de curl natif
  • Débogage binaire difficile
  • Moins accessible (apprentissage)
  • Navigateurs : grpc-web requis
// sensor.proto
message SensorReading {
  string sensor_id = 1;
  float temperature = 2;
  int64 timestamp = 3;
}
// ~123B binaire + ~30B headers HPACK ≈ 153B/req
GraphQL
  • Requêtes flexibles (client-driven)
  • Évite over-fetching / under-fetching
  • Complexité serveur accrue
  • Pas de streaming natif
Non retenu
Message Brokers
Kafka · RabbitMQ · MQTT
  • Asynchrone, découplage fort
  • Replay d'événements (Kafka)
  • Infrastructure lourde à opérer
  • Latence accrue (queue)
Long terme
Verdict préliminaire
Pour SignalWatch, REST vs gRPC est le choix central. Les brokers async sont complémentaires, pas concurrents.

RGPD & Sécurité des données IoT industrielles

Données personnelles en contexte IoT
📍
Localisation du capteur L'emplacement d'un capteur peut identifier un poste de travail, et donc une personne → données personnelles indirectes (RGPD Art. 4)
💓
Données de santé indirectes Capteurs de vibration haute fréquence → possible corrélation avec rythme cardiaque ou fatigue du worker → catégorie spéciale (Art. 9)
🌡️
Patterns comportementaux Séries temporelles de présence/activité → profilage des habitudes de travail → nécessite analyse d'impact (DPIA)
⚠️

Conclusion

Bilan, Limites & Prochaines étapes

Conclusions clés
  • gRPC gagne sur payload (2.7x compact), streaming natif, contrat fort .proto et éco-conception
  • REST gagne sur accessibilité, maintenabilité, écosystème outillage et courbe d'apprentissage
  • Latence loopback : REST légèrement meilleur (0.08ms vs 0.20ms p50) — différence s'atténuera avec latence réseau réelle
  • Architecture hybride recommandée : démarrer REST, planifier migration gRPC pour les flux critiques haute fréquence
  • Rust excellent dans les deux cas — performances natives, sécurité mémoire sans GC
Prochaines étapes
  • Tester avec Docker + réseau bridge pour mesurer l'overhead réseau réel (vs loopback)
  • Activer mTLS mutuel (certificats X.509) pour mesurer l'impact sécurité sur latence
  • Implémenter gRPC server-streaming (StreamSensorReadings) pour flux continu
  • Benchmark avec 1 000+ capteurs simultanés pour valider la scalabilité
  • Intégrer MQTT pour les capteurs à ressources contraintes (ESP32, LoRa)
Limites du benchmark
  • Loopback localhost — latence réseau réelle (1–50ms) changera radicalement les proportions
  • Outils différents — k6 (HTTP/1.1) vs ghz (HTTP/2) → débits non directement comparables
  • Machine unique — Mac de développement, pas d'environnement de production (VM/container)
  • Pas de TLS activé — impact sur performance non mesuré (overhead handshake crypto)
  • Payload minimal — un seul capteur, pas de batch, pas de données volumineuses
  • Pas de garbage collection — Rust favorise les deux cas vs JVM/Go qui auraient des pauses GC
